# 微信公众号封面系统

制作完整文章包、用户要求封面或准备草稿箱时使用。本文件只规定封面用途与交付边界；生图质量、标题融合和裁切安全统一遵循 `visual-quality.md`。

## 默认成品

- `2.35:1` 头条封面：`900×383`，每篇完整文章包必需。
- 正文首图位于 H1 与副标题之后，同样是 `900×383` 图文成品。默认直接复用为头条封面，避免维护两张语义相同的图。
- 只有用户需要不同构图时才单独生成头条封面；独立封面仍使用同一视觉方向并单独完成视觉检查。

正文首图和独立头条封面都必须由生图模型直接生成准确标题与画面融合的成品。脚本只可做尺寸规范化，不能贴字、加白框、黑罩、标签或海报卡片来伪造融合效果。

当前 `draft/add` 流程只通过一个 `thumb_media_id` 发送文章封面。本 Skill 使用 `2.35:1` 成品作为该封面，不暗示方图也被 API 上传。

## 2.35:1 构图

- 只保留一个强视觉想法，禁止拼贴无关符号。
- 准确 H1 只出现一次，位于安静安全区，通常是左侧、左中或平衡的中央。
- 标题、关键主体和关系在手机缩略图下仍然可读。
- 使用暖纸、墨色、低饱和砖红、鼠尾草绿和克制中性色。
- 真实截图、用户图、官方图片、SVG 或 HTML 只能给生成方向提供依据，不能代替最终生成首图。
- 标题错字、多字、缺字、乱码、额外文字或融合生硬时，重新生成或使用生图编辑能力修正。

## 可选 1:1 方图

只有以下任一用途明确存在时制作 `1080×1080` 方图：

- 朋友圈、聊天分享、归档卡片或跨平台复用。
- 多文章列表需要方形缩略图。
- 用户明确要求封面组合。

方图单独构图，使用 4 至 10 个汉字的压缩标题。禁止把 `2.35:1` 盲裁成方图，也不要把完整长标题硬塞进去。普通单篇文章或仅使用本 Skill 的 `draft/add` 时跳过。

## 制作与发布顺序

1. 根据文章主张和视觉隐喻写含准确标题、`2.35:1`、标题安全区、主体安全区与排除项的提示词。
2. 调用真实生图工具生成候选图，逐张查看标题、构图和文字瑕疵。
3. 确定成品并规范为准确 `900×383`；规范化后重新检查标题和主体没有被裁切。
4. 把确定文件登记到 `image-plan.json`，正文用唯一 `data-wlp-visual-id="hero"` 对账。
5. 草稿模式只把已经审查通过的确定文件传给 `publish.ts --cover <path>`。发布命令拒绝临时生图，防止图片绕过视觉 QA。

推荐文件：

```text
cover/wechat-headline-cover.png
cover/wechat-1x1-cover.png    # 仅有真实用途时
```

## 最终检查

- 文件为 `900×383`，比例准确。
- 封面标题与 HTML H1 完全一致，只出现一次。
- 标题融入画面，没有贴字感、额外文字、标志、水印或伪界面。
- 手机缩略图下标题、主体和视觉关系仍然成立。
- 正文首图与单独封面并存时，二者职责或构图确实不同。
- 实际生成方图时使用单独压缩标题，并在完成回报中标为附加素材。
