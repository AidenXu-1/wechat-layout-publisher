# 编辑写作规则

制作预览前先使用本规则。排版再精致，如果正文像没有温度的报告，文章仍然不成立。

## 内容模式边界

- `messy_materials + rewrite`：先区分资料中的事实、观点、经历、数据、重复与冲突，再使用本文件的钩子、人味和段落规则完成定稿。
- `draft_copy + rewrite`：以现有初稿为母版，先诊断，再修订成发布定稿。优化标题、结构、开头、段落、表达、证据连接和结尾，同时保留事实、核心观点、证据边界和作者立场。
- `preserve`：把本文件作为问题扫描清单，只报告不理想之处。不得替换词语、标点、句序或段落顺序；只能在原句边界拆段。缺少标题或副标题时向用户索取，未经授权不得补写。

## 杂乱资料成文流程

`messy_materials` 先找能发动文章的矛盾，再整理章节。禁止把资料按来源顺序改写成报告。

1. **划定事实边界**：分开记录事实、观点、亲历、数字、引语、重复、冲突和待核项。冲突未解决前不得选边补齐。
2. **选择开场锚点**：从真实人物、具体场景、异常数字、意外结果或明确痛点中选一个。没有可感知锚点时，先继续找材料，不用抽象判断硬开头。
3. **确定读者追问**：用一个问题贯穿全文，例如“它为什么能识别我”“这些人为什么同时做出同一种选择”。后续材料必须回答或推进这个问题。
4. **建立叙事骨架**：规划 3 至 8 个不同推进节点。优先使用“异常或场景 → 第一份证据 → 大白话解释 → 新证据或案例升级 → 机制/影响 → 最终判断”。教程类可以改用“痛点 → 做法 → 结果 → 可扩展玩法”的重复单元。
5. **排列证据顺序**：证据紧跟它支持的主张。先让读者看见事实，再解释应该看哪里；不要先下重结论，几屏以后才补证据。
6. **决定显性结构**：并列人物、玩法、步骤使用明确小标题；调查、经历和故事优先让事件自然换场。只在读者可能迷路时增加标题。
7. **延后结论**：让判断由前文证据逐步挣出来。结尾压缩到一个真实判断或人的处境，不重新复述全文。

把 `core_claim`、`reader_question`、`opening_anchor`、`narrative_spine`、`evidence_sequence`、`ending_claim`、`structure_mode` 和 `structure_reason` 写入 `image-plan.json` 的 `editorial_plan`。这些字段必须指向本次材料，禁止填写“引发思考”“层层递进”等空话。

## 初稿文案独立流程

`draft_copy` 已有文章骨架，不从零重写；它仍处于修订阶段，也不能冻结原文。按以下顺序执行：

1. **读取动力线**：标出开头在哪里产生好奇，证据在哪里出现，解释是否过早，哪里重复却没有升级，结尾是否已经被前文支撑。
2. **守住内容边界**：提取不可改变的事实、数字、经历、核心观点、作者立场和来源。无法确认的内容标记待核，不擅自补齐。
3. **记录声音指纹**：记录第一/第二/第三人称、口语浓度、长短句节奏、情绪温度，以及 1 至 6 个必须保留的表达特征。只保护原稿真实存在的特点，不凭空添加粗口、网络梗、错别字或夸张语气。
4. **确定修改优先级**：按“结构与动力 → 证据连接 → 段落呼吸 → 词句准确”处理。先做局部修复；只有动力线断裂、段落无法归位时才整体重构，并在修改说明中明确写出。
5. **逐项修订**：允许重写句段、合并重复和调整章节，但每一处修改都要改善理解、可信度或推进速度，不能只为了显得更漂亮。
6. **定稿自检**：确认标题兑现正文、开头进入具体问题、每一段有职责、证据靠近主张、转折获得下文兑现、结尾给出真实判断。
7. **修改说明**：进入配图前保存主要修改摘要，说明结构、证据、观点表达和声音保护发生了什么变化。

把 `voice_fingerprint` 与 1 至 6 条 `revision_priorities` 一并写入 `editorial_plan`。若初稿实际上只是资料碎片，向用户说明后确认是否改选 A；若用户要求任何字都不改，确认改选 C。

## 内容类型闸门

写作前先分类：

- **新闻 / 事件跟踪**：近期争议、发布、报道、官方回应、社区讨论或媒体覆盖。
- **产品 / 工具介绍**：解释工具、工作流、版本或使用方法。
- **观点评论**：基于已知事实提出判断。
- **知识解释**：讲清概念或机制。
- **经验复盘**：个人工作流、错误、决策或项目回顾。
- **叙事**：以故事推进为主。

新闻或事件跟踪必须有证据图片计划。SVG 解释图或生成的隐喻图不能替代公开来源证据。

### 保守识别新闻

规划视觉前，记录 `content_type`、`classification_confidence` 和 `classification_signals`。

下列信号出现两项以上，或其中一项是文章中心时，将文章归为 `news_event`：

- 近期日期、发布、禁令、争议、政策变化、产品变化、官方回应、公共事件或持续发展中的事件。
- 具名公司、人物、机构或平台做了某事或发布了公告。
- 正文引用或转述官方页面、社交帖子、媒体报道、论文、文档、采访或社区讨论。
- 读者需要确认某条引语、公告、截图、产品状态或公众反应确实存在。
- 使用“今天”“最近”“宣布”“回应”“报道”“爆火”“最新”等时效性词语。

文章从公共事件出发，再加入作者观点或经验时，使用 `mixed_news_commentary`。评论部分不会取消证据要求。

置信度低但存在新闻信号时，也采用 `mixed_news_commentary`，不要归为纯 `opinion`。先收集证据再决定是否展示，比发布一篇没有证据层的新闻型文章更稳妥。

## 新闻 / 事件跟踪规则

满足以下任一情况，优先使用证据截图：

- 文章引用媒体报道、官方回应、产品公告、论文、文档页面、Reddit/X/社区讨论或公共网站。
- 文章描述近期争议、产品变化、账号封禁、政策变化、版本发布或公共讨论。
- 读者需要确认事件确实发生过。

最低要求：

- 完整交付包至少在对应段落附近放一张公开来源截图。
- 来源优先级：官方回应或官方文档，其次媒体报道，再次社区原始讨论。
- 事件源于社交媒体时，能访问原始可识别帖子，就优先于媒体转述。
- 搜索结果页、代码制作的引语卡、重制帖子或生成图都不算证据。
- 因登录墙、网络阻断、Cookie 墙或安全限制无法截图时，在完成回报中说明。
- 先建立证据层，再用 SVG/HTML 图解释机制。

## 标题与读者承诺

标题优先组合“具体对象或场景 + 可核实的异常/冲突/结果 + 读者能得到的认知”。数字、时间、身份和结论必须来自正文与来源。

- 人物群像突出共同选择及其信号，不把名单写成简历目录。
- 经验复盘写清真实投入、变化或结果，再交付方法。
- 争议新闻指出具体行为与影响，避免超过证据的动机指控。
- 教程写清适用对象、数量或入口，正文逐项兑现。
- 社会叙事从一个可见处境进入，再扩大到制度或群体现象。

标题可以有张力，但不得用虚构数字、伪造亲历、无来源的绝对结论或正文没有回答的悬念换点击。

## 开头钩子

`rewrite` 模式先写人能感受到的利害，再解释专业问题，避免以中性报告口吻开场。`preserve` 模式只检查并报告，不改开头。

推荐顺序：

1. 真实感受、经历或读者痛点。
2. 用一句大白话点明冲突。
3. 再进入专业解释。

可参考：

- 具体经历：“半个月被封了三个号以后，我对这件事的感觉变了。”
- 读者痛点：“你以为只是账号风控，后来发现它可能在看你电脑这边的痕迹。”
- 具体不适：“难受的不是封号，是你不知道它到底按什么给你贴标签。”

避免：

- “本文将分析……”
- “这件事不只是……而是……”
- “真正值得关注的是……”
- 读者尚未明白为什么要关心时，就连续抛出四个抽象名词。

## 推进、转折与标题显隐

给每个段落或阅读单元分配一个职责：`hook`、`evidence`、`explanation`、`escalation`、`transition`、`judgment` 或 `release`。相邻段落职责完全重复时，合并、删减或让后者带来新信息。

转折句只在下文会兑现时使用。可以提示“后面还有一层”“这个结果带来新的问题”，但下一段必须立即给出新事实、机制、结果或风险。禁止批量套用“更重要的是”“真正的问题是”“故事才刚刚开始”制造假悬念。

结构选择：

- `visible_sections`：并列人物、工具、步骤、玩法、清单或需要回查的教程。
- `implicit_flow`：调查、亲历、人物故事、社会叙事或靠情绪连续推进的文章。
- `hybrid`：前半段用故事建立问题，后半段用步骤或并列案例交付答案。

显性标题表达真实内容，不使用“背景介绍”“深度分析”“写在最后”占位。隐性结构仍需在 `narrative_spine` 中写清换场节点，不能把“没有小标题”变成没有结构。

## 证据参与叙事

把证据当作文章中的动作，不当作结尾附件：

1. 先提出具体、可验证的主张。
2. 紧接原始截图、数据、引语、产品状态或用户材料。
3. 用一两句指出读者该看哪里，以及证据能证明到什么程度。
4. 再进入机制解释、影响或作者判断。

同一张图同时承担多个主张时拆开说明。证据只能证明它实际显示的内容；作者推断和价值判断另起句子，避免让截图替作者背书。

## 人味闸门

预览前扫描 AI 腔。`rewrite` 模式改成自然的大白话；`preserve` 模式保留原文并在回报中提示。

减少：

- “不只是/不仅是/不仅仅是……而是/也是……”
- “真正的问题是”
- “核心/关键/本质/格局/凸显/反映/至关重要”
- “更成熟的看法是”
- “这件事的重点不是”
- 连续三个抽象概念并列。
- 过多加粗的“金句”。

改写方向：

- 用读者能感受到的场景替代抽象名词。
- 专业术语先用一句大白话解释，再正式使用。
- 允许短句和口语化节奏。
- 段落结尾无需次次上升到“行业意义”。

### 受控的粗粝感

自然表达允许有停顿、短句、口语连接词和轻微重复。修订时保留能证明“这个人确实这样说话”的节奏，但删掉纯粹拖慢理解的赘词。

- 不把所有“我觉得”“然后”“其实”机械删除；判断它是否承担在场感、转折或呼吸。
- 不把每句话都改成对称、完整、正式的书面语。
- 不为了模仿样文主动增加粗口、网络梗、感叹号、错别字或故意不通顺。
- 不复制某个作者的标志性口头禅。学习的是叙事决策、证据顺序和节奏控制。

## 段落密度闸门

一个段落通常只完成一项任务。

`rewrite` 模式下，同一段同时包含以下三项以上时应拆分或重写；`preserve` 模式只能在原句边界拆段：

- 事件背景。
- 技术机制。
- 平台动机。
- 用户风险。
- 行业判断。
- 道德结论。

中文正文建议：

- 超过 90 至 120 个汉字：检查并通常拆分。
- 超过 160 个汉字：除非是刻意保留的叙事段，否则必须拆分。
- 同段出现两个以上抽象概念时，补具体例子或拆分。
- 短段落服务停顿与推进，不追求统一字数。连续多个一句话段落若没有形成节奏或升级，应合并。

## 有深度，也有人味

保持事实边界和技术深度，同时让读者从真实体验进入。

推荐路径：

```text
人能感到的事 -> 大白话解释 -> 证据/截图 -> 机制图 -> 判断
```

避免从这里开始：

```text
行业意义 -> 抽象机制 -> 风险框架 -> 用户情绪
```

## 定稿自检

`rewrite` 进入配图前逐项回答；任何一项只能用空泛形容词回答时，回到正文继续修改：

1. 开头锚定了哪个具体人物、场景、异常、结果或痛点？
2. 读者带着哪个问题继续阅读，正文在哪些节点推进了它？
3. `narrative_spine` 的相邻节点分别增加了什么新信息？
4. 关键主张附近有什么证据，证据与作者推断是否分开？
5. 专业概念是否先用普通人能理解的场景、类比或例子落地？
6. 显性小标题是否真的帮助回查；隐性结构是否仍然清楚换场？
7. `draft_copy` 保住了哪些声音特征，删改了哪些只会拖慢文章的粗糙部分？
8. 结尾的判断由前文哪几项事实支撑，删掉总结性套话后是否仍然成立？
